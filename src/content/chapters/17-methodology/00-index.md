写出能编译的 Rust 代码只是第一步。在真实的工程项目里，你还需要回答：

- 代码应该怎么组织？模块边界在哪里？
- 编码时应该先写什么、后写什么？
- 怎么保证多人协作时代码风格一致？
- 怎么在每次提交时自动检查代码质量？
- 如何找出性能瓶颈并有效优化？

这一章以一个**从零到上线的大工程**为背景，按生命周期顺序讲解 Rust 工程化的核心方法论。

## 学完能做什么

- 从需求出发规划 Rust 工程的模块结构
- 按 Trait 先行的思路有条理地推进编码
- 配置 Clippy、rustfmt、CI 流水线，自动化代码审查
- 用 `cargo audit` 扫描依赖安全漏洞
- 用 `criterion` + flamegraph 定位并量化性能问题

> 这一章的内容偏"工程实践"，不需要背语法，重在理解**为什么要这样做**，建立工程思维。

> **关于方法论的适用性**：本章介绍的流程和工具是 Rust 社区广泛采用的实践，但没有放之四海皆准的方法论。个人项目、小团队快速迭代、大型开源库、嵌入式系统各有侧重——你不一定要把每一步都用上，重要的是理解每种做法**解决了什么问题**，在真实项目中根据情况取舍。
