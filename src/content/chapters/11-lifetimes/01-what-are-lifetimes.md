# 悬垂引用问题

你已经知道 Rust 有"借用"这个概念：可以不转移所有权、只拿一个引用。但引用有个潜在风险——如果被引用的数据已经销毁了，引用还在，就会指向无效内存，这叫**悬垂引用**（dangling reference）。

C/C++ 程序员对这类 bug 再熟悉不过了：use-after-free、野指针……Rust 的目标是让这类错误**在编译期就被发现**，永远不到运行时。

## 一个会出问题的例子

看这段代码（你可能在借用与引用章节已经见过，我们再回顾一下）——它试图在内部作用域之外使用一个指向内部变量的引用：

```rust runnable expect-error
fn main() {
    let r;

    {
        let x = 5;
        r = &x;       // r 借用了 x
    }                 // x 在这里被销毁

    println!("r: {}", r); // 危险！x 已经不存在了
}
```

> Rust 会直接拒绝编译，报错：`` `x` does not live long enough ``

`x` 的生命在内部 `{}` 结束时就结束了，但 `r` 要活到 `println!` 那行。`r` 比它所引用的数据活得更久——这就是悬垂引用。

## 没有问题的版本

只要让被引用的数据比引用活得更久，就没有问题：

```rust runnable
fn main() {
    let x = 5;            // x 在这里创建，活得更长
    let r = &x;           // r 借用 x
    println!("r: {}", r); // 此时 x 还活着，完全合法
}
```

这两个例子的区别只是 `x` 声明的位置，但 Rust 完全知道哪个可以、哪个不行。靠什么知道？靠**借用检查器**。

# 借用检查器

## 编译器如何做判断

Rust 编译器内置了**借用检查器**（borrow checker），它的工作就是比对引用的生命周期与被引用数据的生命周期，确保前者不会超过后者。

我们用注释把生命周期可视化出来，看第一个出错的例子：

```rust
{
    let r;                // ------+-- 'r 的生命周期开始
                          //       |
    {                     //       |
        let x = 5;        // -+--  |  'x 的生命周期开始
        r = &x;           //  |    |
    }                     // -+    |  'x 生命周期结束！x 被销毁
                          //       |
    println!("{}", r);    //       |  r 仍然在用，但 'x 已经结束
}                         // ------+
```

`r` 的生命周期 `'r` 比 `x` 的生命周期 `'x` 更长。`r` 引用了 `x`，所以 `'x` 必须覆盖 `'r` 的整个范围——但它没有，编译器报错。

## 正确例子的生命周期

```rust
{
    let x = 5;            // ------+-- 'x 开始
                          //       |
    let r = &x;           // --+   |  'r 开始
                          //   |   |
    println!("{}", r);    //   |   |
                          // --+   |  'r 结束
}                         // ------+  'x 结束
```

`'x` 完全包含了 `'r`，引用有效，编译通过。

## 生命周期不是程序员"发明"的

生命周期参数（`'a`、`'b` 这样的写法）不是 Rust 独有的概念，它实际上描述的是**引用存在的那段时间**——这段时间本来就存在，只是 Rust 让你在某些场合把它写出来，让编译器能够核验。

就像类型标注一样：变量有类型是客观事实，大多数时候编译器能推断，偶尔你需要写出来。生命周期也是如此——大多数时候编译器能推断（这叫"省略"），偶尔你需要手动标注。

# 练习题

## 基础概念测验

```quiz single
Q: 什么是悬垂引用（dangling reference）？
- 一个可变引用
- 一个指向堆内存的引用
- 一个被多处共享的引用
+ 一个指向已经销毁的数据的引用
E: 悬垂引用是指引用所指向的数据已经被销毁（离开了作用域），但引用本身还在被使用。Rust 的借用检查器在编译期检测并拒绝所有悬垂引用。
```

```quiz single
Q: Rust 中的借用检查器（borrow checker）的主要作用是什么？
- 检查变量是否被正确初始化
+ 比对引用的生命周期和被引用数据的生命周期，确保引用不会指向已销毁的数据
- 检查是否有内存泄漏
- 检查代码中是否有语法错误
E: 借用检查器是 Rust 编译器的一个组件，专门负责分析引用的生命周期。它确保任何引用都不会"活"得比它所指向的数据更久，从而在编译期彻底消灭悬垂引用。
```

```rust
fn main() {
    let r;
    {
        let x = 10;
        r = &x;
    }
    println!("{}", r);
}
```

```quiz single
Q: 上面这段代码的结果是？
- 能编译，输出 10
- 能编译，但运行时 panic
- 能编译，输出未定义值
+ 不能编译，x 的生命周期不够长
E: 这段代码无法编译。x 在内部 {} 结束时就被销毁了，但 r 还要活到 println! 那行。借用检查器发现 r 的生命周期超过了 x 的生命周期，报错 "`x` does not live long enough"。
```

```quiz multi
Q: 下列哪些说法关于生命周期是正确的？
+ 生命周期检查发生在编译期，不影响运行时性能
- 生命周期标注会改变引用实际存活的时间
- 只有函数参数才有生命周期
+ 大多数情况下，编译器可以自动推断生命周期（无需手动标注）
+ 每个引用都有一个生命周期，表示引用保持有效的时间段
E: 生命周期是引用保持有效的时间段，这是客观存在的。生命周期标注只是帮助编译器理解你的意图，不会改变实际的存活时间。Rust 提供生命周期省略规则，让大多数情况不需要手动标注。所有检查在编译期完成，零运行时开销。
```

```quiz single
Q: 如果引用 `r` 的生命周期为 `'r`，被引用的数据 `x` 的生命周期为 `'x`，想让代码合法，需要满足什么条件？
+ `'x` 必须至少和 `'r` 一样长（即 `'x` 覆盖 `'r` 的整个范围）
- `'r` 和 `'x` 必须完全相等
- 没有要求，只要不用 unsafe 都是安全的
- `'r` 必须比 `'x` 更长
E: 被引用的数据必须比引用活得更久——或者至少一样久。只要 `'x` 覆盖了 `'r` 的整个使用范围，引用就始终有效。这就是借用检查器的核心判断逻辑。
```
